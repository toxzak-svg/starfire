## Core Insight

**Hard problems don't compress well.** We figured this out during research discussions about the high-compression hallucination problem. The better a compressor gets, the more it selects for things that fit its biases — not genuine understanding.

**New inversion:** Use compressibility as the *routing signal*. Not a learned router that can collapse — an external signal that's non-gameable.

---

## Architecture: Compressibility-Aware Routing (CAR)

```
Token
  │
  ▼
┌──────────────┐
│  SSM State   │◀── what does the model EXPECT to see?
│  Predictor   │
└──────┬───────┘
       │
       ▼
  Prediction Error
       │
       ├─── low error ──▶ CHEAP PATH (small MLP) — "I knew this"
       │
       └─── high error ──▶ EXPAND (large base model) — "this is new"
```

### Why this fixes router collapse

Learned routers collapse because they're trained to minimize loss — easiest path wins. Collapse is equilibrium.

Prediction error is **external to the model**. It can't be gamed. The SSM state makes a prediction. The error is just: "did the state predict this token well?" That's a ground truth signal.

### The state as compressor

The SSM state doesn't just *store* context — it *predicts* what comes next. Lower prediction error = better compression = better model.

Hard tokens (high error) get routed to the expensive path. Easy tokens (low error) get the cheap path. The model learns to compress better (improving the state predictor) AND to allocate compute where it matters.

### Human cognition analogy

Familiar task (driving a known route) → cheap path, habit, low attention.  
Novel event (pedestrian jumps out) → working memory spikes, full reasoning.

Surprise = high prediction error = expand.

## Memory Math for 100-150B on P5000

| Component | Memory | Where |
|---|---|---|
| Base model 100B @ 4-bit | 50GB | CPU/NVMe streaming |
| Working state (persistent) | ~1GB | GPU SRAM |
| Activations (BS=1, seq=2048) | ~4GB | GPU |
| Optimizer (trainable params only) | ~8GB | GPU |
| **Total** | **~63GB** | needs optimization |

**Optimization path:**
- Tensor parallelism across 2 P5000s → 31GB per GPU
- Or: 50B params instead of 100B → 32GB with aggressive optimization
- Or: use gradient checkpointing + CPU optimizer offload for the base model

**Key architectural insight:** The base model is 4-bit quantized and never fully on GPU. We stream shards from NVMe, async, hidden behind compute. The "expansion" copies activations from the streamed base, not weights.

## Why This Could Work

1. **Non-collapsible routing signal** — prediction error can't be gamed by the model
2. **Real improvement signal** — better state = lower error = fewer expansions = faster inference
3. **Complementary to Nue** — Nue solves "which path per layer." CAR solves "when to go big." They compose.
4. **Scales to hardware constraints** — streaming base + small working state fits P5000

---

## What We Need to Prove

1. SSM state can predict well enough to route correctly 80%+ of the time
2. The expansion decisions are learnable (not random)
3. Quality at 100B effective compute matches 100B dense
