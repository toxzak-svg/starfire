# H4 real closed-cycle shadow transfer

## Status

Frozen CPU acceptance probe for latent-concept transfer over real Starfire component attempts that are independently judged and applied through the closed cognitive cycle.

This experiment remains shadow-only. No eligible concept can modify the live CHARGE router.

## Question

Can Starfire start from one visible `Custom("unresolved")` CHARGE class, observe independently judged attempts from its real cognitive components, induce opaque distinctions from CHARGE-native residual geometry, freeze those distinctions, and retain useful resolver discrimination on later unseen task families under exact matched-budget controls?

## Observation path

The probe does not inject generated resolver outcome tables.

For every task:

1. a real Starfire subsystem emits CHARGE:
   - `MetaCognition` knowledge gap
   - `PredictionCenter` refutation contradiction
   - `QuanotTrajectoryEmitter` reservoir trajectory residual
2. the human-authored `ChargeKind` is hidden from ontology induction by replacing it with `Custom("unresolved")`
3. deterministic CHARGE-native residual geometry is prepended to the raw residual:
   - RMS
   - mean absolute value
   - non-zero fraction
   - positive fraction
   - maximum absolute value
4. each real candidate component runs through its existing Starfire API:
   - `ReasoningEngine`
   - SQLite `Store::search_memories`
   - `CausalEngine`
   - `PredictionCenter`
   - `MetaCognition`
5. the component output is submitted as an action to an `Environment` implementation with a deterministic target verifier
6. environment progress becomes an `OutcomeWitness`
7. `RelativeImprovementJudge` caps the resolver-requested discharge by measured objective improvement
8. `CognitiveCycleState::apply_judgment` applies the accepted discharge to a fresh counterfactual pending cycle
9. only cycle-applied accepted discharge is recorded in `OntologyObservation`

The reusable bridge is `cognitive_cycle::CycleObservationRecorder`.

Each resolver is replayed from the same initial CHARGE snapshot. This produces a counterfactual outcome matrix for shadow evaluation without allowing the first attempted resolver to change the state seen by later candidates.

## Why residual geometry is allowed

The feature adapter is deterministic and label-blind. It sees only the numeric residual vector.

It does not inspect:

- original `ChargeKind`
- scope names
- emitter identity
- hidden task class
- resolver names
- resolver outcomes
- target text

The same adapter must be applied before historical fitting and future routing.

The purpose is to let ontology search discover stable distributional structure such as sparse versus dense residuals without relying on one exact raw residual coordinate remaining semantically aligned across unrelated topic families.

## Controlled component fixture boundary

As in the earlier real-component CHARGE probe, knowledge fixtures are intentionally placed in the subsystem that natively owns them:

- empirical facts in SQLite `Store`
- contradiction-correcting facts in `ReasoningEngine` memories
- causal edges in `CausalEngine`

This tests whether latent routing distinctions recover functional specialization of actual Starfire components under closed-cycle judging. It does not claim the components learned those facts autonomously.

## Frozen window protocol

The executable is:

```text
cargo run -p star --example h4_real_cycle_shadow_probe --locked
```

The final constants are fixed in source before the first acceptance run.

Window structure:

- 2 training windows
- 1 independent promotion holdout window
- 4 future transfer windows
- 36 observations per window
- 12 surface variants for each of 3 real emitter classes
- 5 real resolver attempts per observation

Training families:

- DNS / copper / cache miss
- mutex / sound / heavy rain

Promotion holdout family:

- photosynthesis / moonlight / insufficient coolant

Future families:

- compiler / borrow checker / packet loss
- mitochondria / glass / low pipe pressure
- HTTP 404 / seasons / combustion
- database index / bats / friction

Exact task semantics in the promotion and future windows do not appear in training.

## Matched-budget controls

The shadow candidate's proposal budget is its actual `candidates_considered` count from empirical induction. Its routing budget is the exact number of future observations.

Every control must report exactly the same proposal and routing counts or `ShadowPromotionMonitor` returns a budget-mismatch error.

### Matched random partition search

Each proposal is a complete hash partition with the same effective routing-group count as induced concepts plus parent fallback. Resolver leaders are learned from training outcomes. The strongest training proposal receives one holdout gate.

### Matched permuted-feature policy search

Every residual dimension and persistence value is independently permuted while independently judged resolver outcomes remain attached to observations.

The control receives the exact proposal budget and searches complete multi-concept policies from the same threshold vocabulary. The strongest training policy receives one holdout gate.

## Frozen final gates

The acceptance executable exits nonzero unless all pass:

1. all observations originate from the three real subsystem-backed CHARGE emitters
2. every visible kind presented to ontology induction is `Custom("unresolved")`
3. every observation contains five non-zero-cost outcomes produced through `OutcomeWitness -> RelativeImprovementJudge -> CognitiveCycleState`
4. at least 2 concepts are promoted from historical windows
5. aggregate future shadow efficiency is at least `1.25x` the undifferentiated baseline
6. shadow beats the baseline in `100%` of future windows
7. the worst future window retains at least `1.10x` baseline efficiency
8. future shadow efficiency is at least `1.20x` every exact matched-budget control

The seed, task families, feature adapter, and gate thresholds must not be modified after observing the first acceptance result.

## Claim boundary after a pass

A pass would support only:

> Over a controlled stream built from real Starfire emitters, real Starfire cognitive component outputs, independent environment feedback, the independent discharge judge, and persistent cognitive-cycle accounting, a frozen latent-concept ontology induced from historically undifferentiated CHARGE can retain useful resolver discrimination on unseen task families and outperform exact matched-budget random and feature-destroyed controls.

A pass would not establish:

- live automatic concept promotion is safe
- the components autonomously learned their fixture knowledge
- open-world concept induction
- recursive ontology growth
- relation induction or reification
- arbitrary distribution-shift stability
- human-like semantics
- AGI
- consciousness

The next pressure after a pass is to record naturally occurring resolver attempts from a running closed-cycle environment rather than a controlled component fixture matrix.
