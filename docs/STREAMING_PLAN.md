# Bonsai-8B Streaming Implementation Plan

> Auto-generated by Marble during self-improvement heartbeat (2026-04-08).

## Problem

`generate_impl` in `lib/llm/mod.rs` is **blocking** — it runs the full autoregressive loop, accumulating all tokens in a `Vec<u32>`, and only returns the complete text once generation finishes.

For a 100-token response at ~50ms/step, the user waits ~5 seconds with no feedback.

## Solution: Callback-Based Streaming

Add a streaming interface that **yields tokens as they're generated**, before the full response is complete.

```rust
/// Generate with streaming callback — yields tokens one at a time.
/// Callback is called after each new token is produced.
pub fn generate_stream<F>(
    &mut self,
    prompt: &str,
    max_new_tokens: usize,
    temperature: f64,
    top_p: Option<f64>,
    mut callback: F,
) -> AnyhowResult<()>
where
    F: FnMut(&str) -> bool,  // returns false to abort early
```

The callback receives each decoded token string as it's generated. Return `false` to stop early.

## API Changes

### 1. Add `generate_stream` to `LlmEngine`

```rust
pub fn generate_stream<F>(
    &mut self,
    prompt: &str,
    max_new_tokens: usize,
    temperature: f64,
    top_p: Option<f64>,
    callback: F,
) -> AnyhowResult<()>
where
    F: FnMut(&str) -> bool,
{
    let token_ids = self.tokenize(prompt)?;
    if let Some(bos) = self.bos_token_id {
        token_ids.insert(0, bos);
    }

    let seed = std::time::SystemTime::now()
        .duration_since(std::time::UNIX_EPOCH)
        .map(|d| d.as_secs())
        .unwrap_or(42);
    let mut logits_processor = LogitsProcessor::new(seed, Some(temperature), top_p);

    let eos = self.eos_token_id.unwrap_or(0);
    let mut all_tokens = Vec::new();
    let prompt_len = token_ids.len();

    for step in 0..max_new_tokens {
        let index_pos = prompt_len.saturating_sub(1) + step;

        let input_ids: Vec<u32> = if step == 0 {
            token_ids.clone()
        } else {
            if let Some(&last) = token_ids.last() {
                token_ids.push(last);
            }
            vec![*token_ids.last().unwrap_or(&0)]
        };

        let input_tensor = Tensor::new(input_ids.as_slice(), &self.device)?
            .reshape(&[1, input_ids.len()])?;
        let logits = self.model.lock().unwrap()
            .forward(&input_tensor, index_pos)?;

        let last_logits = logits.squeeze(0)?.squeeze(0)?;
        let next_token = logits_processor.sample(&last_logits)? as u32;

        if next_token == eos && eos != 0 {
            break;
        }

        // Decode + stream
        let token_str = self.decode_token(next_token);
        if !token_str.is_empty() {
            if !callback(&token_str) {
                break;  // caller aborted
            }
        }

        token_ids.push(next_token);
        all_tokens.push(next_token);
    }

    Ok(())
}
```

## Integration Points

### Runtime `chat()` → streaming polish

Currently:
```rust
fn chat(&mut self, input: &str) -> String {
    let reasoning = self.knowledge_graph.query(input);
    let rough = self.voice.speak(&reasoning);
    let polished = self.llm.polish(&rough);  // blocking
    polished
}
```

With streaming:
```rust
fn chat_stream<F>(&mut self, input: &str, sink: F) -> AnyhowResult<()>
where
    F: FnMut(&str) -> bool + Send + 'static,
{
    let reasoning = self.knowledge_graph.query(input);
    let rough = self.voice.speak(&reasoning);
    // Stream directly to caller instead of buffering full polished text
    self.llm.polish_stream(&rough, sink)
}
```

### Where streaming shines

1. **Voice integration**: Each token fed to TTS engine as it arrives — user hears words progressively instead of waiting 3-5s
2. **Long outputs**: 500+ token responses stream immediately instead of blocking
3. **Abort on context limit**: Caller can stop generation early if response is going too long

## Open Questions

1. **Should polish() also stream?** Yes — polish() is called in the critical path for user-facing output
2. **Callback or Iterator?** Closure/callback is simpler in sync Rust; Iterator requires async. Callback is fine for now.
3. **Backpressure**: Currently no flow control. If caller is slow (e.g., TTS lag), generation continues anyway.
4. **Non-blocking**: The forward pass is CPU-bound, not async. Full non-blocking would need a separate thread + channel.
